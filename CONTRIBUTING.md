## Development Setup

### Prerequisites

- Python 3.8 or higher
- pip (Python package manager)
- Git

### Installation

1. Clone the repository:
   ```bash
   git clone https://github.com/your-org/flowlang.git
   cd flowlang
   ```

2. Create a virtual environment:
   ```bash
   python -m venv venv
   source venv/bin/activate  # On Windows: venv\Scripts\activate
   ```

3. Install in development mode with all dependencies:
   ```bash
   pip install -e ".[dev]"
   ```

4. Verify the installation:
   ```bash
   flowlang --version
   pytest --version
   ```

### ANTLR4 Grammar (Optional)

If you need to modify the grammar:

1. Install ANTLR4 tool (Java required):
   ```bash
   # macOS
   brew install antlr

   # Ubuntu/Debian
   sudo apt-get install antlr4

   # Or download manually from https://www.antlr.org/download.html
   ```

2. Grammar files are located in `grammar/` (not included in Phase 1)

3. Regenerate parser (when grammar changes):
   ```bash
   antlr4 -Dlanguage=Python3 -visitor -listener FlowLangLexer.g4 FlowLangParser.g4
   ```

│   ├── flow_node.py
│   ├── decorators.py
│   └── qos.py

**Important**: Files in `flowlang/grammar/` are auto-generated by ANTLR4. Do not edit them manually. Modify the grammar files (`.g4`) and regenerate instead.

## Running Tests

### Run All Tests

```bash
pytest
```

### Run with Coverage

```bash
pytest --cov=flowlang --cov-report=term-missing
```

### Run Specific Test Categories

```bash
# Unit tests only
pytest -m unit

# Integration tests only
pytest -m integration

# Exclude slow tests
pytest -m "not slow"

# Specific test file
pytest tests/test_ast.py

# Specific test class
pytest tests/test_ast.py::TestSourceLocation

# Specific test method
pytest tests/test_ast.py::TestSourceLocation::test_location_creation
```

### Test Markers

Tests are marked with the following categories:

- `unit`: Fast unit tests (default)
- `integration`: Integration tests requiring more setup
- `slow`: Tests that take significant time to run

## Code Style

We use the following tools to maintain code quality:

### Black (Code Formatting)

```bash
# Format all files
black flowlang/ tests/

# Check formatting without making changes
black --check flowlang/ tests/
```

Configuration: `pyproject.toml` (default Black settings)

### mypy (Type Checking)

```bash
# Type check all files
mypy flowlang/

# Show more detailed errors
mypy --show-error-codes flowlang/
```

Configuration: `pyproject.toml` or `mypy.ini`

### flake8 (Linting)

```bash
# Run linter
flake8 flowlang/ tests/
```

Configuration: `setup.cfg` or `.flake8`

### Pre-commit Hook (Recommended)

Install pre-commit to run checks automatically:

```bash
pre-commit install
pre-commit run --all-files
```

### Style Guidelines

- **Docstrings**: Use Google-style docstrings for all public functions, classes, and modules
- **Type Hints**: Add type hints to all function signatures
- **Line Length**: 88 characters (Black default)
- **Imports**: Group imports (stdlib, third-party, local) with blank lines between

Example docstring:

```python
def example_function(param1: str, param2: int) -> bool:
    """Short description of the function.

    Longer description if needed. Explain the purpose and
    any non-obvious behavior.

    Args:
        param1: Description of first parameter
        param2: Description of second parameter

    Returns:
        Description of return value

    Raises:
        ValueError: When input is invalid

    Example:
        >>> example_function("test", 42)
        True
    """
    return True
```

## Pull Request Process

1. **Create a Branch**
   ```bash
   git checkout -b feature/your-feature-name
   # or
   git checkout -b fix/issue-description
   ```

2. **Make Your Changes**
   - Write code following the style guidelines
   - Add tests for new functionality
   - Update documentation as needed

3. **Run Quality Checks**
   ```bash
   # Format code
   black flowlang/ tests/

   # Type check
   mypy flowlang/

   # Run linter
   flake8 flowlang/ tests/

   # Run tests
   pytest
   ```

4. **Commit Your Changes**
   ```bash
   git add .
   git commit -m "feat: add your feature description"
   ```

5. **Push and Create PR**
   ```bash
   git push origin feature/your-feature-name
   ```

6. **PR Requirements**
   - PR title follows conventional commits format
   - Description explains what changed and why
   - All CI checks must pass
   - Code review approval required
   - Tests added for new functionality

### PR Title Format

```
type(scope): description

Examples:
- feat(codegen): add C++ generator
- fix(semantic): resolve symbol lookup in nested scopes
- docs(readme): update installation instructions
- test(security): add test for eval detection
```

Types:
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Code style changes (formatting, etc.)
- `refactor`: Code refactoring
- `test`: Test additions or updates
- `chore`: Build process or auxiliary tool changes

## Commit Messages

We follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

```
type(optional scope): description

[optional body]

[optional footer]
```

Examples:

```
feat(semantic): add type inference for list types

Implements type inference for list<type> declarations.
Currently supports primitive and qualified types.

Closes #123
```

```
fix(codegen): correct parameter default value generation

Fixed bug where None default values were not properly handled
in generated Python code.
```

## Reporting Issues

### Security Issues

Please do not report security issues publicly. Email security concerns to: security@your-org.com

### Bug Reports

When reporting bugs, please include:

1. **Description**: Clear description of the bug
2. **Steps to Reproduce**: Minimal example that demonstrates the issue
3. **Expected Behavior**: What you expected to happen
4. **Actual Behavior**: What actually happened
5. **Environment**:
   - Python version
   - OS version
   - FlowLang version

Example:

```markdown
**Description**
Parser fails on nodes with empty parameter blocks.

**Steps to Reproduce**
1. Create test.node with:
   ```
   node TestNode {
       parameters {
       }
   }
   ```
2. Run `flowlang test.node`

**Expected Behavior**
Should parse successfully with empty parameters.

**Actual Behavior**
Parser throws IndexError.

**Environment**
- Python 3.10.4
- Ubuntu 22.04
- FlowLang 0.1.0
```

### Feature Requests

When requesting features:

1. Describe the use case
2. Explain why current functionality is insufficient
3. Provide examples of desired syntax/behavior
4. Reference similar features in other tools if applicable

## Code Review Process

All submissions require review before merging:

1. Automated checks must pass (tests, linting, type checking)
2. At least one maintainer approval required
3. Review focuses on:
   - Correctness
   - Test coverage
   - Documentation
   - Security implications
   - Performance considerations

## Release Process

Releases follow Semantic Versioning:

- **MAJOR**: Breaking changes to language syntax or generated code
- **MINOR**: New features, backward compatible
- **PATCH**: Bug fixes, backward compatible
